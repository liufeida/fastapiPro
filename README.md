# fastapi-demo

一个以 `FastAPI` 为后端核心的练手项目，当前目标是先把一套清晰、可维护、可扩展的 Web API 基础设施搭起来，再逐步把 AI 相关能力系统化接入进来。

这个项目不只是写几个接口，更像是一条完整的学习和落地路线：

- 先把传统后端能力打牢：接口设计、认证鉴权、文件上传、数据库建模、数据访问分层、迁移管理。
- 再把 AI 技术栈一块一块补上：`Prompt`、`LangChain`、`RAG`、`Skill`、`MCP Server` 等。
- 最终形成一个既能做业务接口、又能承载 AI 能力编排的完整后端项目。

## 当前技术栈

项目当前后端技术选型如下：

- `FastAPI[standard]`
- `uv`
- `SQLModel`
- `PostgreSQL`
- `Alembic`
- `asyncpg`

这套组合的目标很明确：

- `FastAPI[standard]` 负责 API 开发体验、异步能力、文档生成与基础工程化。
- `uv` 负责依赖管理和运行环境，保持项目启动和安装流程简洁。
- `SQLModel` 负责数据模型与类型友好的 ORM 开发体验。
- `PostgreSQL` 负责关系型数据存储，也为后续接入 `pgvector` 做 RAG 准备。
- `Alembic` 负责数据库迁移，保证模型演进可控。

## 项目定位

这个仓库的定位不是“一次性 demo”，而是一个可以持续演进的后端实验田：

- 第一阶段，做成一套结构清晰的 FastAPI 后端模板。
- 第二阶段，把 AI 能力逐步接进现有服务层，而不是另起一套割裂的代码。
- 第三阶段，沉淀成一个可以复用的 AI Backend Playground。

换句话说，这个项目未来既可以服务传统 CRUD，也可以服务 AI 应用。

## 当前目录思路

目前项目已经具备比较典型的分层结构，方向是对的：

```text
app/
├─ api/           # 路由层
├─ core/          # 核心配置、数据库、鉴权等
├─ models/        # 数据模型
├─ repository/    # 数据访问层
├─ schemas/       # 请求/响应结构
├─ services/      # 业务逻辑层
├─ utils/         # 工具函数
└─ main.py        # 应用入口
```

这个结构很适合继续演进，因为它天然支持下面几件事：

- 把 AI 能力放到 `services/ai/` 或 `app/ai/` 下做独立封装。
- 把 RAG 的索引、召回、重排拆成单独模块。
- 把工具调用、技能系统、MCP Server 能力做成可插拔组件。

## AI 学习与落地路线

下面这条路线建议按顺序推进，不要一上来就直接做 Agent。先把基础打稳，再做复杂编排。

### Phase 0：把后端地基打稳

目标：先把传统后端部分做扎实，让后面的 AI 能力有可靠载体。

建议完成：

- 配置管理：环境变量、开发/测试/生产配置拆分。
- 数据库管理：连接配置、迁移流程、初始化流程规范化。
- 用户体系：注册、登录、JWT、权限边界。
- 文件体系：上传、访问、元数据记录。
- 基础工程能力：日志、异常处理、统一响应结构、测试。

阶段产出：

- 一套可运行、可迁移、可测试的 FastAPI 项目骨架。
- 一套稳定的 `API -> Service -> Repository -> DB` 分层习惯。

### Phase 1：先学 Prompt，不急着上框架

目标：先理解“大模型是怎么被正确调用的”，而不是先堆工具。

建议完成：

- 学会写角色设定、任务约束、输出格式约束。
- 学会把 Prompt 模板参数化，而不是把提示词写死在代码里。
- 学会做结构化输出，比如 JSON Schema / Pydantic 输出约束。
- 学会区分系统提示、开发者提示、用户输入的职责。
- 学会做基础评估：什么提示词更稳定，什么输出更容易失控。

阶段产出：

- `prompt/` 或 `app/ai/prompts/` 目录。
- 一套可复用的 Prompt 模板管理方式。
- 1 到 2 个稳定的小能力，比如：
  - 文本摘要
  - 标题生成
  - 问答分类
  - 信息抽取

### Phase 2：引入 LangChain，但只解决“工程组织”问题

目标：把模型调用、Prompt、输出解析、工具调用包装起来，减少重复代码。

这一阶段不要把 `LangChain` 当魔法，而是把它当“AI 工程组织工具”。

建议完成：

- 封装统一的大模型调用入口。
- 使用 Prompt Template 管理输入变量。
- 使用 Output Parser 管理结构化输出。
- 初步接触 Chain 设计，把单步任务组合起来。
- 了解 Tool Calling 的基础模式。

阶段产出：

- `app/ai/llm/`
- `app/ai/prompts/`
- `app/ai/chains/`
- `app/ai/parsers/`

到这一步，你应该能把“AI 调一次接口”升级为“AI 能稳定完成一个具体任务”。

### Phase 3：做第一个 RAG 闭环

目标：让模型不只会“生成”，还会“基于你的知识库回答”。

建议按这个顺序做：

1. 文档接入：支持上传文档或导入知识内容。
2. 文档切分：研究 chunk 大小、重叠策略。
3. 向量化：选择 embedding 模型。
4. 向量存储：优先考虑 PostgreSQL + `pgvector`。
5. 检索：实现 top-k 召回。
6. 生成：把召回内容拼进 Prompt。
7. 评估：检查幻觉、召回偏差、答案覆盖率。

建议能力边界：

- 先做单知识库问答。
- 再做多文档问答。
- 最后再做带权限控制的企业知识库。

阶段产出：

- `app/ai/rag/loader.py`
- `app/ai/rag/splitter.py`
- `app/ai/rag/embedder.py`
- `app/ai/rag/retriever.py`
- `app/ai/rag/pipeline.py`

这一阶段是整个 AI 项目真正开始“有业务价值”的分水岭。

### Phase 4：把 Skill 思维引入系统

目标：把 AI 能力从“一个大而全的 Agent”拆成多个边界清晰的小能力模块。

这里的 `Skill` 可以理解为：

- 一个明确目标
- 一套固定输入
- 一套固定 Prompt / Tool / Output
- 一套可复用的执行逻辑

例如你可以设计这些技能：

- `summary_skill`：摘要技能
- `qa_skill`：知识库问答技能
- `extract_skill`：结构化信息抽取技能
- `rewrite_skill`：改写润色技能
- `search_skill`：检索增强技能

阶段产出：

- `app/ai/skills/`
- 每个 skill 单独定义输入、输出、依赖工具、执行器
- skill 注册表和调度入口

做到这里，系统会开始具备“能力组件化”的味道，后面再接 Agent 或 MCP 会顺很多。

### Phase 5：引入 Tool / Agent 能力，但要克制

目标：让模型不只会回答，还能“调用系统能力”。

建议从最小工具集开始：

- 用户查询工具
- 文件检索工具
- 知识库检索工具
- 数据统计工具
- 外部接口查询工具

注意这里的原则：

- 能不用 Agent 就别上 Agent。
- 能单步完成就不要多步规划。
- 能固定流程就先别做自主决策。

阶段产出：

- 一套内部 Tool 协议
- AI 调用业务函数的统一封装
- 基础 Agent 执行链路

这一阶段的重点不是“酷炫”，而是“可控”。

### Phase 6：实现 MCP Server

目标：把这个项目里的能力标准化暴露出去，让外部 AI 客户端可以调用。

你可以把 `MCP Server` 理解为：给大模型或 AI 客户端提供标准工具接口的一层协议服务。

建议这个阶段做的事情：

- 梳理哪些能力适合作为 MCP Tool 暴露。
- 设计清晰的工具输入输出。
- 保持工具粒度小、职责单一。
- 做好鉴权、限流、日志和错误处理。
- 优先暴露高价值工具，而不是一口气全开放。

优先级高的 MCP 能力示例：

- 查询用户信息
- 查询文件列表
- 搜索知识库
- 触发一个指定 skill
- 获取某个任务执行结果

阶段产出：

- `app/mcp/`
- MCP Tools 注册机制
- 面向 AI 客户端的标准化能力出口

做到这里，项目就不只是“一个后端”，而是“一个可被 AI 系统调用的平台能力层”。

### Phase 7：补齐生产级能力

目标：从“能跑”走向“能长期维护”。

建议补齐：

- 可观测性：日志、链路追踪、耗时统计。
- 成本控制：token 统计、缓存、限流。
- 安全：Prompt Injection 防护、敏感信息过滤、权限隔离。
- 评估：RAG 命中率、回答质量、失败案例回放。
- 测试：接口测试、服务测试、Prompt 回归测试。

这一步很重要，因为 AI 系统一旦进入真实场景，工程问题往往比模型问题更先暴露。

## 推荐的实际推进顺序

如果你想一步一步来，我建议按下面的顺序推进：

1. 完成基础用户系统、文件系统、数据库迁移规范。
2. 增加统一配置、日志、异常处理中间件。
3. 做一个最小 AI 接口，比如“文本总结”或“内容抽取”。
4. 把 Prompt 模板化、结构化输出跑通。
5. 接入 LangChain，把模型调用逻辑组织起来。
6. 基于 PostgreSQL 演进到 `pgvector`，完成第一版 RAG。
7. 把 AI 能力拆成多个 skill。
8. 给 skill 和检索能力加 tool 化封装。
9. 最后再做 MCP Server，对外提供标准能力。

这个顺序的好处是：

- 每一步都能看到成果。
- 每一步都能单独验证价值。
- 不会因为一开始目标太大而把工程做乱。

## 未来目标形态

理想状态下，这个项目会演进成下面这个样子：

- 传统后端部分负责用户、权限、文件、数据管理。
- AI 基础层负责模型调用、Prompt 管理、输出解析。
- RAG 层负责知识接入、切分、向量化、召回、生成。
- Skill 层负责能力组件化和业务编排。
- MCP Server 层负责对外暴露标准化 AI 工具能力。

最后形成一个兼具以下特征的项目：

- 能做常规业务接口
- 能做 AI 能力封装
- 能做知识库问答
- 能做工具调用
- 能对外提供标准 AI 接口

## 一句话总结

这是一个以 `FastAPI[standard] + uv + SQLModel + PostgreSQL` 为起点，逐步把 `Prompt`、`LangChain`、`RAG`、`Skill`、`MCP Server` 全链路做一遍的后端成长型项目。

核心策略不是一口气把 AI 全堆进去，而是按阶段推进，每一步都留下清晰的工程结构和可复用能力。
